# Fiction Reader Onboarding Harness

## 목적

이 문서는 독자가 모르는 개념을 장면 안으로 들여오는 방법을 정한다. 목표 독자는
전문가가 아니다. 배경지식 없이 읽어도 누가 무엇을 잃고, 왜 지금 선택해야 하는지
알아야 한다.

단, 기술 설명 자체를 금지하지 않는다. QFUDS SAGA는 기술 개념이 재미의 일부다.
문제는 설명이 아니라 **설명이 장면의 손실과 연결되지 않는 것**이다.

## Boundary

fiction/provenance only. 연구 증거, 투자 조언, 법률 조언, 암호 보안 조언이 아니다.
새 외부 source claim 없음.

## 기본 규칙

```text
정확한 설명 -> 쉬운 비유 -> 손실 상황 -> 선택 압력
```

독자가 이해해야 하는 것은 완전한 이론이 아니라 장면에서 필요한 최소 작동 방식이다.
다만 최소 작동 방식은 틀리면 안 된다. 개발자, 연구자, 크립토 독자가 봐도 핵심
메커니즘은 맞아야 하고, 일반 독자는 그 메커니즘을 생활 비유와 손실 상황으로 붙잡을
수 있어야 한다.

## 기술 설명 3단 통과 기준

짧은 강의노트급 설명은 허용한다. 단, 아래 세 단계를 모두 통과해야 한다.

| 단계 | 통과 기준 |
| --- | --- |
| 1. 정확한 기술 설명 | 공개키/개인키/서명/장부/해시/합의가 실제로 무엇인지 짧고 정확하게 설명한다. 전문가 독자가 봐도 틀리지 않아야 한다 |
| 2. 초등학생도 이해할 수 있는 생활 비유 | 주소, 열쇠, 도장, 공책, 지문처럼 독자가 이미 아는 물건으로 붙잡게 한다. 기술과 완전히 같지 않으면 `비유상`임을 드러낸다 |
| 3. 장면 압력 | 이 개념 때문에 인물이 지금 무엇을 잃는지, 어떤 버튼을 눌러야 하는지, 누르면 왜 망하고 안 눌러도 왜 망하는지 연결한다 |

설명 뒤 1-3문단 안에는 손실, 비용, 행동이 나와야 한다. 설명이 끝난 뒤 독자는
“그래서 이 사람이 지금 뭘 누르면 망하는지” 말할 수 있어야 한다.

실패:

```text
공개키는 타원곡선 암호에서...
```

통과:

```text
공개키는 공개되어도 되는 검증 정보다. 평소에는 공개키를 알아도 개인키를 계산할 수 없다.
비유상 공개키는 집 주소이고, 개인키는 문을 여는 열쇠다.
Q-Day 이후에는 주소만 보고 열쇠를 복사할 수 있게 된다.
그래서 지금 전송하면 공개키가 장부에 남고, 그 순간 도망칠 주소를 사냥꾼들에게 보여 주는 일이 된다.
```

## 독자 수준 가정

- 기술 배경 없음.
- Bitcoin을 투자자산 정도로만 알거나 아예 모를 수 있음.
- 공개키/개인키/서명/해시/합의/블록체인을 구분하지 못함.
- 그래도 구체적 손실, 버튼, 돈, 신원, 법적 효력, 시간 제한은 이해함.

## 온보딩 단위

| 단위 | 설명 |
| --- | --- |
| Plain handle | 어려운 개념을 붙잡는 짧은 한국어 손잡이 |
| Technical name | 보존할 원어 또는 기술어 |
| Accurate technical note | 전문가 독자가 봐도 틀리지 않는 최소 작동 설명 |
| Simple analogy | 일반 독자가 붙잡을 수 있는 생활 비유. 필요하면 `비유상` 표시 |
| Scene object | 화면, 버튼, 도장, 양식, 금고, 장부 같은 물건 |
| Loss translation | 그 개념을 모르면 인물이 무엇을 잃는가 |
| Required action | 인물이 지금 해야 하는 행동 |
| Failure if misunderstood | 독자가 헷갈리면 장면 긴장이 죽는 지점 |

## 설명해야 하는 것과 보여줘도 되는 것

| 분류 | 처리 |
| --- | --- |
| 장면 선택을 이해하는 데 필요한 개념 | 한 문장 최소 정의 + 장면 물건 + 손실로 설명 |
| 세계관 깊이를 주는 개념 | 용어는 보존하되 첫 등장은 짧게, 자세한 설명은 뒤로 미룸 |
| 전문가가 알면 좋은 개념 | 각주나 bible에 두고 prose에서는 압력만 사용 |
| 플롯을 멈추는 개념 | 장면 밖 design 문서로 보냄 |

## SF 온보딩 방식

기술·제도·역사 개념이 많은 SF는 한 방식만 쓰면 리듬이 무너진다. 작업 단위마다
아래 방식 중 무엇을 쓰는지 먼저 고른다.

| 방식 | 설명 | 강점 | 위험 |
| --- | --- | --- | --- |
| 튜토리얼형 | 독자처럼 모르는 인물이 배운다 | 접근성이 높다 | 설명용 인물이 될 수 있다 |
| 문제 발생형 | 고장, 피해, 악용을 먼저 보여 주고 규칙을 나중에 이해시킨다 | 긴장감이 세다 | 독자가 너무 오래 모르면 혼란스럽다 |
| 짧은 강의노트형 | 기술 자체를 짧고 정확하게 설명한다 | 정확성과 지적 쾌감이 산다 | 장면 리듬을 끊을 수 있다 |
| 은유 보조형 | 생활 비유로 먼저 붙잡게 한다 | 직관적이다 | 비유가 기술을 왜곡할 수 있다 |
| 인월드 문서형 | 로그, 법정 기록, 감사 보고서, 매뉴얼, 경고문으로 설명한다 | 세계관과 제도감을 동시에 준다 | 과하면 위키처럼 읽힌다 |
| 반복 노출형 | 처음엔 다 설명하지 않고 여러 장면에서 의미를 쌓는다 | 세련되고 자연스럽다 | 초반 진입 장벽이 생긴다 |
| 전문가 대화형 | 이미 아는 사람들이 논쟁하며 필요한 설명을 흘린다 | 지적 긴장과 갈등이 생긴다 | 독자를 위한 대화처럼 보이면 죽는다 |

기술 자체가 재미인 작품은 짧은 강의노트형을 피하지 않는다. 단독으로 쓰지 말고
문제 발생형, 은유 보조형, 인월드 문서형 중 하나와 붙인다.

## QFUDS SAGA 기본 선택

QFUDS SAGA의 기본 온보딩 방식은 **테크노스릴러형 + 인월드 문서형**이다.

```text
기술 설명은 정확하게 한다.
곧바로 위기, 손실, 버튼, 비용으로 연결한다.
제도와 세계관은 감사 로그, 시스템 경고, seal protocol, court exhibit,
Last Archive fieldmark 같은 인월드 문서로 분산한다.
```

운영 기준:

- 기본형: 테크노스릴러형. 기술 설명은 위기·손실·선택과 붙인다.
- 보조형: 인월드 문서형. 제도, 선례, 감사 기록, 프로토콜은 문서 조각으로 처리한다.
- 은유는 보조다. 정확한 기술 설명을 대체하지 않는다.
- 튜토리얼형은 제한적으로만 쓴다. 설명용 인물을 만들지 않는다.
- 반복 노출형은 `seal`, `claim`, `recoverable`, `verification` 같은 제도어에 쓴다.

## 금지 방식

- 독자가 아는 척해야만 장면이 작동하는 것.
- technical term만 늘어놓고 손실이 없는 것.
- 개념을 비유로만 말하고 실제 작동 방식을 숨기는 것.
- 설명이 끝났는데 인물의 선택지가 바뀌지 않는 것.
- “이건 복잡하지만 대충 위험하다”로 넘기는 것.

## 리뷰 질문

장면마다 아래를 묻는다.

| 질문 | 통과 기준 |
| --- | --- |
| 기술 설명이 정확한가 | 전문가 독자가 봐도 공개키/서명/장부/합의 등의 핵심 작동이 틀리지 않음 |
| 쉬운 비유가 있는가 | 일반 독자가 주소/열쇠/도장/공책/지문 같은 손잡이로 이해할 수 있음 |
| 독자가 이 용어를 몰라도 위기를 이해하는가 | 예: 돈/신원/권리/시간 제한으로 말할 수 있음 |
| 기술 설명이 버튼이나 행동으로 이어지는가 | 설명 뒤 인물이 뭔가 누르거나 멈추거나 숨김 |
| 설명이 재미를 죽이지 않는가 | 정보가 새 위험을 열거나 선택지를 바꿈 |
| 비유가 실제 기술을 가리지 않는가 | 비유 앞뒤에 최소 작동 방식이 있고, 완전히 같지 않으면 비유임을 알림 |
| 다음 장면에서 같은 설명을 반복하지 않는가 | 첫 등장은 손잡이, 이후는 행동으로만 상기 |

## SAGA 적용

SAGA에서 암호/비트코인/복원 개념은 강점이다. 설명을 숨기지 않는다. 대신 모든 설명은
아래 중 하나로 이어져야 한다.

- 돈이 움직이지 않음.
- 신원이 증명되지 않음.
- 서명이 위조되거나 무효화됨.
- 법적 소유와 실제 접근이 갈라짐.
- 복원 가능성이 청구권으로 오해됨.
- 검증자가 마지막 판정권을 얻음.
